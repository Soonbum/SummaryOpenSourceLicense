# 오픈소스 라이선스 요약

* 이 글은 Gemini를 이용하여 작성한 글을 정리한 내용입니다.
  - 다음 내용은 대표적인 내용만 기재한 것임을 참고 바랍니다.

| 라이선스 | 소스 코드 공개 범위 | 상업적 이용 | 배포 시 의무 사항 | 특허 관련 조항 | 대표 사례 |
| -------- | ------------------- | ----------- | ----------------- | -------------- | --------- |
| AGPL | 서버 실행만 해도 전체 공개 | 가능 | 네트워크 서비스 제공 시 소스 공개 (실행만 해도 공개 의무) | GPL v3 기반 보호 | MongoDB(구), Grafana |
| GPL | 전체 공개 (파생 저작물 포함) | 가능 | 복제/수정/배포 시 소스 공개 (라이브러리를 포함시키지 않고 별도 프로세스로 실행할 경우 공개하지 않아도 됨) | v3부터 명시적 보호 | Linux 커널, Git |
| LGPL | 라이브러리 수정 시에만 공개 | 가능 | 정적 연결 시 소스/오브젝트 공개 (dll, so 사용시 공개하지 않아도 됨) | GPL v3 기반 보호 | FFmpeg, glibc |
| MPL | 수정된 파일만 공개 | 가능 | 결합된 다른 파일은 공개 불필요 | 명시적 보호 | Firefox, LibreOffice |
| Apache | 공개 의무 없음 | 가능 | 수정한 파일에 대한 고지 의무 | 강력한 특허 보장 | Kubernetes, Android |
| MIT | 공개 의무 없음 | 가능 | 저작권 및 라이선스 문구 유지 | 없음 (포괄적 허용) | React, Node.js |

* 주요 차이점 핵심 요약
  - 공개 범위의 차이 (전염성)
    * AGPL/GPL: 가장 강력합니다. 이 라이선스가 포함된 코드를 활용하면 프로젝트 전체가 같은 라이선스로 묶여 소스를 공개해야 합니다.
    * LGPL/MPL: 중간 단계입니다. '내가 건드린 오픈소스 파일'만 공개하고, 내가 새로 만든 독자적인 파일은 비공개로 유지할 수 있는 길을 열어줍니다.
    * Apache/MIT/BSD: 매우 자유롭습니다. 소스 코드를 공개하지 않고 결과물만 판매하는 것이 가능합니다.
  - 클라우드 서비스(SaaS)와 AGPL
    * 과거 기업들이 GPL 코드를 서버에서만 돌리고 배포는 하지 않는 방식으로 공개 의무를 피해가자, 이를 막기 위해 나온 것이 AGPL입니다. 웹 서비스만 운영해도 소스를 공개해야 하므로 기업에서 가장 주의 깊게 살펴봅니다.
  - 기업이 Apache를 선호하는 이유
    * MIT는 매우 간결하지만 '특허'에 대한 명시적 언급이 없습니다. 반면 Apache 2.0은 소프트웨어에 포함된 특허권을 사용자에게 허용한다는 조항과, 사용자가 특허 소송을 제기할 경우 라이선스를 종료시킨다는 보복 조항이 있어 법적 방어력이 훨씬 뛰어납니다.

* 라이선스 선택 시 팁
  - 완전 독점 소프트웨어를 만들고 싶다면: MIT, Apache를 선택하거나 이 라이선스들로 구성된 오픈소스만 사용하세요.
  - 커뮤니티와 함께 성장하고 싶다면: GPL, MPL 등을 고려하세요.
  - 라이브러리 배포를 통해 표준이 되고 싶다면: LGPL이나 MIT가 유리합니다.

* 주의사항
  - 내가 직접적으로 사용하는 라이브러리가 MIT 라이선스라 하더라도 그 라이브러리가 GPL 라이선스인 다른 코드를 사용한다면 내가 만든 프로그램도 GPL의 영향을 받게 됩니다. (라이브러리 배포자가 실수하거나 제대로 고지하지 않을 수 있으므로 라이선스 체크가 필요함)
  - 라이브러리의 라이선스는 버전이 바뀜에 따라 얼마든지 배포자가 변경할 수 있으나 소급 적용되지는 않습니다. (1.0에서 MIT 라이선스가 적용되어 있었는데 2.0에서 GPL 라이선스로 변경될 경우 계속 1.0을 사용하면 공개 의무 지키지 않아도 됨)

# 그 외 라이선스에 대해

* 세상에는 수천 개의 오픈소스 라이선스가 존재하며, 이를 모두 나열하는 것은 사실상 불가능에 가깝습니다. 하지만 **OSI(Open Source Initiative)**라는 공인 단체에서 인증한 라이선스만 해도 100여 개가 넘습니다. 가장 대중적인 것들을 제외하고, 특정 상황이나 기업에서 독자적으로 만들어 실무에서 마주칠 확률이 있는 라이선스들을 성격별로 분류하여 나열해 드립니다.

**1. 전염성이 강한 라이선스 그룹 (Copyleft)**
  * GPL 계열과 비슷하게 소스 코드 공개 의무가 강한 것들입니다.
    - EUPL (European Union Public License): 유럽연합에서 만든 라이선스로, GPL과 호환되면서도 유럽 법률에 최적화되어 있습니다.
    - CPAL (Common Public Attribution License): 서비스 초기 화면에 원저작자의 로고나 이름을 반드시 표시해야 한다는 '어트리뷰션' 조항이 강화된 GPL 계열입니다.
    - EPL (Eclipse Public License): 이클립스 재단에서 만든 라이선스입니다. GPL보다는 약간 유연하지만, 수정한 부분에 대한 공개 의무가 있습니다.

**2. 기업들이 만든 전용 라이선스 (Business/Source Available)**
  * 최근 클라우드 기업들과의 전쟁으로 인해 탄생한 "무늬만 오픈소스"인 경우가 많습니다. (SaaS 개발 시 가장 주의해야 함)
    - SSPL (Server Side Public License): MongoDB가 만든 라이선스로, 소스 공개 범위가 인프라 전체로 확장되어 사실상 독점이나 다름없습니다.
    - BSL (Business Source License): MariaDB, Terraform 등이 사용합니다. 일정 기간이나 특정 조건(매출액 등)하에서는 유료지만, 시간이 지나면 오픈소스로 전환되는 독특한 구조입니다.
    - RSAL (Redis Source Available License): 레디스가 최근 도입한 라이선스로, 경쟁사가 상용 서비스를 하는 것을 금지합니다.

**3. 매우 자유로운 라이선스 그룹 (Permissive)**
  * MIT, Apache처럼 제약이 거의 없는 부류입니다.
    - BSD (Berkeley Software Distribution) 계열:
      * 0-Clause (0BSD): 저작권 고지조차 필요 없는, 거의 '공유 도메인' 수준의 자유도입니다.
      * 2-Clause / 3-Clause: MIT와 거의 흡사하며 저작권 고지 범위에 차이가 있습니다.
    - ISC (Internet Systems Consortium): MIT와 법적으로 동일한 효력을 가지지만 문구가 훨씬 짧고 간결합니다.
    - Unlicense: 저작권자가 자신의 권리를 완전히 포기하고 공공의 영역(Public Domain)으로 던진 라이선스입니다.

**4. 기타 특수 라이선스**
  * Artistic License: Perl 언어 생태계에서 주로 사용하며, 원본과 수정본의 구분을 엄격히 요구합니다.
  * WTFPL (Do What The F*ck You Want To Public License): "네 마음대로 해라"라는 이름의 파격적인 라이선스입니다. 법적 효력보다는 상징적인 의미가 큽니다.
  * Microsoft Public License (MS-PL): 마이크로소프트가 만든 오픈소스 라이선스로, 소스 공개 의무가 배포 시에만 발생합니다.

* 한눈에 보는 라이선스 스펙트럼
  - 주의할 점: 모든 라이선스를 다 외울 필요는 없습니다. 실무에서는 "GPL 계열인가?"와 "기업 전용(SSPL/BSL)인가?" 이 두 가지만 먼저 필터링해도 리스크의 90%를 막을 수 있습니다.

| 분류 | 라이선스 예시 | 주요 특징 |
| ---- | ------------- | --------- |
| Public Domain | Unlicense, 0BSD | 아무런 제약 없음 (가장 자유로움) |
| Permissive | MIT, Apache, BSD, ISC | 저작권 고지만 하면 됨 |
| Weak Copyleft | LGPL, MPL, EPL | 수정된 파일/라이브러리만 공개 |
| Strong Copyleft | GPL, AGPL, EUPL | 프로젝트 전체 소스 코드 공개 |
| Source Available | SSPL, BSL | 상업적 이용이나 클라우드 서비스 제한 |
