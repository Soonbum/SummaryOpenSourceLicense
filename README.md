# SummaryOpenSourceLicense

* 오픈소스 라이선스에 대한 요약: 이 글은 Gemini를 이용하여 작성한 글을 정리한 내용입니다.

| 라이선스 | 소스 코드 공개 범위 | 상업적 이용 | 배포 시 의무 사항 | 특허 관련 조항 | 대표 사례 |
| -------- | ------------------- | ----------- | ----------------- | -------------- | --------- |
| AGPL | 서버 실행만 해도 전체 공개 | 가능 | 네트워크 서비스 제공 시 소스 공개 (실행만 해도 공개 의무) | GPL v3 기반 보호 | MongoDB(구), Grafana |
| GPL | 전체 공개 (파생 저작물 포함) | 가능 | 복제/수정/배포 시 소스 공개 (라이브러리를 포함시키지 않고 별도 프로세스로 실행할 경우 공개하지 않아도 됨) | v3부터 명시적 보호 | Linux 커널, Git |
| LGPL | 라이브러리 수정 시에만 공개 | 가능 | 정적 연결 시 소스/오브젝트 공개 (dll, so 사용시 공개하지 않아도 됨) | GPL v3 기반 보호 | FFmpeg, glibc |
| MPL | 수정된 파일만 공개 | 가능 | 결합된 다른 파일은 공개 불필요 | 명시적 보호 | Firefox, LibreOffice |
| Apache | 공개 의무 없음 | 가능 | 수정한 파일에 대한 고지 의무 | 강력한 특허 보장 | Kubernetes, Android |
| MIT | 공개 의무 없음 | 가능 | 저작권 및 라이선스 문구 유지 | 없음 (포괄적 허용) | React, Node.js |

* 주요 차이점 핵심 요약

  - 공개 범위의 차이 (전염성)

    * AGPL/GPL: 가장 강력합니다. 이 라이선스가 포함된 코드를 활용하면 프로젝트 전체가 같은 라이선스로 묶여 소스를 공개해야 합니다.

    * LGPL/MPL: 중간 단계입니다. '내가 건드린 오픈소스 파일'만 공개하고, 내가 새로 만든 독자적인 파일은 비공개로 유지할 수 있는 길을 열어줍니다.

    * Apache/MIT/BSD: 매우 자유롭습니다. 소스 코드를 공개하지 않고 결과물만 판매하는 것이 가능합니다.

  - 클라우드 서비스(SaaS)와 AGPL

    * 과거 기업들이 GPL 코드를 서버에서만 돌리고 배포는 하지 않는 방식으로 공개 의무를 피해가자, 이를 막기 위해 나온 것이 AGPL입니다. 웹 서비스만 운영해도 소스를 공개해야 하므로 기업에서 가장 주의 깊게 살펴봅니다.

  - 기업이 Apache를 선호하는 이유

    * MIT는 매우 간결하지만 '특허'에 대한 명시적 언급이 없습니다. 반면 Apache 2.0은 소프트웨어에 포함된 특허권을 사용자에게 허용한다는 조항과, 사용자가 특허 소송을 제기할 경우 라이선스를 종료시킨다는 보복 조항이 있어 법적 방어력이 훨씬 뛰어납니다.

* 라이선스 선택 시 팁

  - 완전 독점 소프트웨어를 만들고 싶다면: MIT, Apache를 선택하거나 이 라이선스들로 구성된 오픈소스만 사용하세요.

  - 커뮤니티와 함께 성장하고 싶다면: GPL, MPL 등을 고려하세요.

  - 라이브러리 배포를 통해 표준이 되고 싶다면: LGPL이나 MIT가 유리합니다.

* 주의사항

  - 내가 직접적으로 사용하는 라이브러리가 MIT 라이선스라 하더라도 그 라이브러리가 GPL 라이선스인 다른 코드를 사용한다면 내가 만든 프로그램도 GPL의 영향을 받게 됩니다. (라이브러리 배포자가 실수하거나 제대로 고지하지 않을 수 있으므로 라이선스 체크가 필요함)
 
  - 라이브러리의 라이선스는 버전이 바뀜에 따라 얼마든지 배포자가 변경할 수 있으나 소급 적용되지는 않습니다. (1.0에서 MIT 라이선스가 적용되어 있었는데 2.0에서 GPL 라이선스로 변경될 경우 계속 1.0을 사용하면 공개 의무 지키지 않아도 됨)
